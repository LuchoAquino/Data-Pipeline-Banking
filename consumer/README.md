# Consumer Module

## Code Description and Flow

This script (`kafka_to.py`) is a **Kafka consumer** designed for a banking data pipeline project. It consumes Change Data Capture (CDC) events from Apache Kafka (generated by Debezium from PostgreSQL database changes) and batches them into Parquet files for upload to MinIO (an S3-compatible object storage). This enables real-time data streaming from transactional databases to analytical storage, supporting ETL processes with tools like DBT or Airflow.

### Key Purpose
- **Data Ingestion**: Captures inserts/updates from `customers`, `accounts`, and `transactions` tables.
- **Transformation & Storage**: Converts JSON events to structured Parquet format with date partitioning (e.g., `customers/date=2026-04-14/`).
- **Efficiency**: Uses batching (50 records) to optimize uploads and reduce costs.
- **Integration**: Part of a full-stack setup with Docker (Kafka, Postgres, MinIO), allowing scalable data warehousing.

### Flow Overview
1. **Initialization**:
   - Load environment variables (Kafka brokers, MinIO credentials) from `.env`.
   - Set up KafkaConsumer to subscribe to Debezium topics (`banking_server.public.*`).
   - Initialize boto3 client for MinIO and ensure the target bucket exists.

2. **Batch Processing Setup**:
   - Define `write_to_minio()` function: Converts records to Pandas DataFrame, writes to local Parquet, uploads to MinIO with timestamped keys, and cleans up.
   - Initialize buffers (dicts) for each topic to accumulate records.

3. **Consumption Loop**:
   - Poll Kafka for messages in an infinite loop.
   - For each message: Extract topic, parse Debezium JSON payload, and get the "after" record (post-change data).
   - Append valid records to the topic's buffer and log for debugging.
   - When buffer reaches batch size (50), call `write_to_minio()` to upload the batch and reset the buffer.

4. **Output**:
   - Files stored in MinIO with partitioning for easy querying (e.g., via Athena or Spark).
   - Console logs confirm uploads and debug incoming records.

To run, ensure Kafka, Debezium, and MinIO are up via `docker-compose`. The script runs continuously, processing events as they arrive.
